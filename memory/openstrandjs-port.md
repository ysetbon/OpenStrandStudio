We are porting OpenStrand Studio (PyQt5) to the web as **OpenStrandJS**, a fidelity-first rewrite living in the sibling folder `C:\Users\YonatanSetbon\projects\OpenStrandJS` (separate git repo, not a clone). Stack: TypeScript + Paper.js (for QPainterPath-equivalent boolean ops/stroking) + Canvas2D in a real browser, React for UI later. Method: translate the Qt source faithfully and verify every piece against the running Qt app via a golden-image pixel diff.

**Build order is renderer-before-UI** (the user's explicit call): scaffold + verification harness → reference-render (Qt) vs js-render diff over the `output/`/sample corpus → port rendering (basic strands → masked boolean strands → shadows) → tune one knot with `/loop` → only then build React UI.

Non-obvious environment facts (cost real time to discover):
- The project `.venv` is **broken** — it was created on another machine (`E:\githubFiles`, user "Setbon") and its base interpreter path doesn't exist. Do NOT use it.
- Use `OpenStrandStudio/src/build_env/Scripts/python.exe` instead — Python 3.12.4 with PyQt5 5.15.2 + PIL, works headless.
- Headless `canvas.render(painter)` renders **blank** because `_paintEventInner` with `use_supersampling=True` (default, `supersampling_factor=2`) paints to an internal `render_buffer`, never the passed painter. The existing `json_to_png_exporter` is broken for the same reason. Fix: bypass widget rendering and replicate `_paintEventInner`'s draw loop directly — iterate `canvas.strands`, call `strand.draw(painter, skip_painter_setup=True)`, pre-drawing an attached strand's parent once for z-order. See `OpenStrandJS/tools/reference_render.py`.
- Saved files use the `OpenStrandStudioHistory` wrapper (`states[]` with steps); extract `states[step].data` (flat `{strands, groups}`) before `load_strands`. The generated `output/mxn_lh_*.json` files are an OLD schema (no `attached_to`) and currently fail to load.

The reference oracle (`tools/reference_render.py`) is proven: it renders the single-strand and full three-strand-braid fixtures correctly (over-under masking included), emitting a PNG + `meta.json` (exact image size + content offset) with no auto-crop, so the JS side can diff deterministically.

The full measurement loop works end-to-end: `tools/js_render.mjs` (Paper.js in Chromium via Playwright) → `tools/diff.mjs` (pixelmatch) → `tools/prove.mjs` orchestrates reference→js→diff. Scores (all 4 fixtures): `single_strand` **99.68%**, `three_strand_braid` **97.85%**, `overhand_knot` **98.10%**, `closed_knot` **98.80%**. Masking + curve port + shadows all done; the renderer generalizes across braid and knots with no per-fixture code. **Shadows are ported** (`draw_strand_shadow` + `draw_mask_strand_shadow`), gated on `meta.shadow_enabled` (per-fixture; canvas default is off — the braid has shadows OFF, the knot ON). `js_render.mjs` must `process.exit(0)` (Playwright hangs after screenshot otherwise — this stalled chained diffs/bg tasks). The remaining residual is now (1) the ~1px stroked-outline band on every strand (Qt QPainterPathStroker vs sampled offset) and (2) junction/closure caps where strand segments join (`has_circles`, visible at the closed_knot bottom). `OpenStrandJS/CONTINUATION.md` has the full roadmap/scores.

Renderer state (`web/render.html`): faithful port of `strand.py::_build_curve_profile` (2-segment cubic incl. third control point); real stroke→outline (sample centerline, offset ±width/2, fill body + pen the boundary incl. caps; `resolveCrossings()` to clean concave folds) — this is the masking primitive; supersampling matched to the app by rendering offscreen at NxN and downscaling in-page (Playwright's canvas screenshot grabs the backing store, so CSS downscale doesn't work). Masking ported from `masked_strand.py`: parse MaskedStrand layer_name "a_b_c_d" → first=a_b (top), second=c_d (bottom); mask = first.stroked(width) ∩ second.stroked(width+2*stroke+4) − deletion_rectangles; redraw top clipped to mask. Draw strands in list order.

CRITICAL non-obvious finding: the curve parameters `control_point_base_fraction`, `distance_multiplier`, `curve_response_exponent` are **canvas-level settings, NOT in the saved JSON**. For these fixtures they are **1.0 / 2.0 / 2.0** (not the `getattr` fallbacks 0.4/1.2/1.5 in `_build_curve_profile`). Using the wrong values makes every curved strand's control points diverge badly (visible as solid-red mismatch on long tails). `reference_render.py` now exports them from `canvas.strands[0]` into `meta.curve_params`, and `render.html` reads `meta.curve_params`. Verified: with 1.0/2.0/2.0 the JS control points for strand 3_6 match Qt's path elements exactly.

Diagnostic tip: dump Qt's actual cubic control points with `path.elementAt(i)` (not just `pointAtPercent`) to compare against the JS `buildProfile` output — that's how the curve-param bug was found.

Known remaining gaps: a small concave-fold spike on tight S-curves (cosmetic, masking-safe); the ~1px stroked-outline band along every strand (Qt QPainterPathStroker vs sampled-offset — the expected irreducible residual); shadows at crossings not yet ported (`draw_mask_strand_shadow`/`draw_strand_shadow` in `shader_utils.py`); only 3 fixtures. Background render commands' stdout often doesn't flush to the task file — write results to a file and read that. Playwright needs the `chromium-headless-shell` package (separate from `chromium`).

Editor-specific non-obvious findings: (1) paper.js auto-applies `window.devicePixelRatio` — at DPR≠1 (display zoom) it double-scales on top of the renderer's own supersampling and the drawing lands at the wrong size. Fix: set `hi.setAttribute('hidpi','off')` on the offscreen canvas (paper honors it → pixelRatio 1). Harness-safe (Playwright runs DPR=1). (2) The renderer paints via `requestAnimationFrame`, which is throttled in a backgrounded tab — headless probes saw a blank 300×150 canvas until a screenshot foregrounded it; store mutations are synchronous so assert on store state, not pixels. (3) Playwright `chromium.launch()` hangs in this env (and `networkidle` never fires against the Vite HMR socket) — use claude-in-chrome against the user's real Chrome instead, and dispatch PointerEvents at transform-computed CLIENT coords (MCP screenshots are scaled NON-uniformly vs the real client, so eyeballed pixel targets miss). (4) MaskedStrands serialize `control_points` as `[null,null]`; `manual_circle_visibility` rides the `extra` passthrough bag so absent stays absent. Dev-only `window.__store` / `window.__io` debug handles (guarded by `import.meta.env.DEV`).

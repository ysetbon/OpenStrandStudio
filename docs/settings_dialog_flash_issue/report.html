🐛 Settings Dialog Position Flash

Investigated 2026-07-19 · OpenStrandStudio v1.109 · Windows 11, PyQt5 · reproduced with an instrumented run (repro.py in this folder)

Symptom

Opening the Settings dialog (main window → ⚙) briefly shows the dialog at the wrong position and wrong size. Roughly a third of a second later it visibly shrinks and jumps to its final centered position. The flash happens on the first open and on every re-open.

Measured timeline

The user-visible flash is the ≈280–350 ms window between “Show” and the late “Resize”/“Move”, during which the dialog sits on screen at the wrong geometry.

Root cause

1 — Primary: heavy work runs in showEvent before Qt can position the dialog

src/settings_dialog.py : 6859, SettingsDialog.showEvent()

def showEvent(self, event):
    if hasattr(self.parent_window, 'language_code'):
        self.current_language = self.parent_window.language_code
        self.update_translations()      # ← ≈300 ms: retranslates EVERY widget,
        self.update_layout_direction()  #   rebuilds the button-guide HTML, etc.

    super().showEvent(event)            # ← QDialog.showEvent → adjustPosition()
                                        #   (the built-in "center over parent")
                                        #   runs ~350 ms too late

Qt centers a QDialog over its parent inside QDialog::showEvent() (via adjustPosition()). By the time this override calls super().showEvent(), the native window has already been mapped on screen — so the centering executes as a visible jump instead of an invisible pre-placement. The same retranslation also changes the layout's size hints, producing the visible size pop a moment earlier. Note the retranslation runs on every single open, even though the language almost never changes between opens.

2 — Contributing: setWindowFlags() on every open recreates the window

src/main_window.py : 658, MainWindow.open_settings_dialog() (also in the RuntimeError recovery path at line 676)

self._settings_dialog.setWindowFlags(
    self._settings_dialog.windowFlags() & ~Qt.WindowContextHelpButtonHint)
self._settings_dialog.show()

In Qt, changing window flags destroys and recreates the native window (the logged WinIdChange). This discards the window manager's placement and guarantees the first mapping happens at stale coordinates. Stripping the "?" help button is a one-time construction concern, not a per-open one.

3 — Nothing positions the dialog before show()

open_settings_dialog() resizes the dialog (once) but never moves it. The final position is only ever established by Qt's late centering described above — so the first mapped frame is always wrong.

How to fix (proposal — not yet applied)

#ChangeWhereEffect
1 Strip Qt.WindowContextHelpButtonHint once, in the constructor; delete both per-open setWindowFlags(...) calls. settings_dialog.py __init__
main_window.py 658, 676		 No more native-window recreation (WinIdChange) on open.
2 Retranslate only when the language actually changed: 
if self.current_language != self.parent_window.language_code:
    self.current_language = ...
    self.update_translations()
    self.update_layout_direction()
 settings_dialog.py 6859 Removes ≈300 ms of work from every open; kills the size pop in the common case.
3 Pre-position before showing: in open_settings_dialog(), before show(), activate the layout and center the dialog on the main window with an explicit move(). An explicit move sets WA_Moved, so Qt's own late adjustPosition() is skipped and the window maps directly at its final spot. main_window.py 656–661 Dialog appears at the correct position in the very first frame; no jump.
4 (Optional hardening) Wrap any remaining heavy refresh in setUpdatesEnabled(False) / (True) so intermediate layout states can never paint. settings_dialog.py showEvent Defense in depth against future slow work added to showEvent.
